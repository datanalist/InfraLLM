## 1. Концепция в одном абзаце

**InfraLLM** — это инфраструктурная LLM-платформа («LLM Gateway»), которая принимает унифицированные запросы на генерацию по OpenAI-совместимому API, умно маршрутизирует их между несколькими LLM-провайдерами (реальный Anthropic + мок-сервисы), управляет реестром A2A-агентов и собирает полную телеметрию по латентности, токенам и стоимости. Цель — продемонстрировать понимание ключевых строительных блоков LLM-инфраструктуры: балансировки, наблюдаемости, реестра, guardrails и аутентификации.

## 2. Проблема и почему сейчас

LLM-вызовы — дорогой и переменный ресурс. Прямое подключение приложений к одному провайдеру создаёт ряд проблем:
- **Vendor lock-in:** смена провайдера требует переписывания приложения.
- **Нет отказоустойчивости:** падение провайдера = падение функционала.
- **Нет наблюдаемости:** невозможно понять, куда уходят деньги и время.
- **Нет контроля:** prompt-injection и утечка секретов проходят насквозь.
- **Нет учёта мульти-агентов:** A2A-взаимодействия не стандартизированы.

InfraLLM решает это в виде единой точки входа с маршрутизацией, телеметрией, реестром и фильтрами.

## 3. Целевой пользователь

- **Первичный:** разработчик/архитектор, строящий мульти-LLM или мульти-агентное приложение. Использует InfraLLM как gateway.
- **Вторичный:** автономный AI-агент (A2A), которому нужно найти другого агента или сделать LLM-вызов через стандартизированную точку.
- **В контексте курса:** оценочная комиссия ИТМО LLM Track.

## 4. Цели проекта и метрики успеха

| # | Цель | Метрика успеха |
|---|---|---|
| G1 | **Демонстрабельность** | `docker-compose up` поднимает весь стек < 60 сек; demo-сценарий проходит без ручных правок |
| G2 | **Качество архитектуры** | Архитектурный документ с диаграммами C4/sequence; обоснование каждого ключевого решения |
| G3 | **Глубина технических решений** | В отчёте: latency-routing vs round-robin сравнение на цифрах; обоснование выбора стека |
| G4 | **Наблюдаемость** | Дашборды Grafana с реальными живыми метриками: RPS, p50/p95, TTFT, TPOT, $/req, ошибки по провайдерам |
| G5 | **Устойчивость** | Под нагрузкой 100 RPS с одним «упавшим» провайдером система продолжает обслуживать запросы (с деградацией, без падения) |
| G6 | **Безопасность (L3)** | 5+ типовых prompt-injection / secret-leak паттернов блокируются guardrail-фильтром |

## 5. Scope

### Внутри scope

| Уровень | Включено | Приоритет |
|---|---|---|
| **L1** | OpenAI-compat вход; маршрутизация по `model`; round-robin/weighted; streaming SSE; health-checks; OTel→Prometheus→Grafana; 2 мок-провайдера + Anthropic | P0 (must) |
| **L2** | **Google A2A v0.1 MVP** (Agent Card + JSON-RPC tasks/send/get/cancel/sendSubscribe); динамический реестр провайдеров (CRUD); latency-based и health-aware routing; circuit breaker; TTFT/TPOT/token/cost метрики; MLflow tracing | P0 (must) |
| **L3** | Guardrails (минимум: prompt-injection + secret-leak); token-based auth; нагрузочный тест с отчётом | P1 (should — MVP-уровень) |

### Вне scope

- ❌ Реальный OpenAI API (нет ключа) — только Anthropic + моки
- ❌ Multi-tenancy с биллингом
- ❌ Distributed cache, distributed rate limiting
- ❌ K8s, Helm, HA-развёртывание
- ❌ Веб-UI для управления реестром (только REST API)
- ❌ Долгосрочное хранение трасс (только текущая сессия + MLflow)
- ❌ Fine-grained RBAC (только bearer-token = разрешён/нет)
- ❌ Поддержка >2 одновременных провайдеров с балансировкой между разными моделями
- ❌ Cost-based routing (трекаем стоимость, но не маршрутизируем по ней)

## 6. Реалистичность сроков — открытый риск

**Заявленный scope: L1+L2+L3 за 1-2 недели соло.** Это **высокий риск**. Грубая оценка трудозатрат:

| Блок | Дни (опт.) | Дни (песс.) |
|---|---:|---:|
| L1 (proxy + 2 mocks + compose + OTel + Prom + Grafana) | 3 | 6 |
| L2 (registry + smart routing + MLflow + token/cost) | 4 | 7 |
| **L2 A2A MVP** (Google A2A v0.1 subset: AgentCard + JSON-RPC + 4 tasks methods) | **1.5** | **2.5** |
| L3 (guardrails + auth + 2 load scenarios + отчёт) | 3 | 6 |
| Документация, диаграммы, защита | 2 | 4 |
| **Итого** | **13.5** | **25.5** |

**Митигации:**
1. **Жёсткий timeboxing** на каждый уровень. Если L1 не готов за 4 дня — стоп, упрощаем.
2. **L3 — обязательно MVP-уровень:** один guardrail-движок (`llm-guard` готовый), простой bearer-token (не JWT), **2 нагрузочных сценария** (steady + provider-failure).
3. **A2A — MVP-подмножество**, не полный протокол: 4 метода JSON-RPC, без push notifications, без file artifacts, без resumable streams.
4. **Использовать готовое:** `litellm` как provider-SDK, `prometheus-fastapi-instrumentator` для метрик, `llm-guard` для guardrails, `locust` для нагрузки.
5. **Fallback plan:** если в день 8 видим, что L3 не успеваем — фиксируем L1+L2 в полном объёме и в L3 делаем только guardrails + минимальный load report.

**Точечная экономия времени под A2A (даёт +1 день к L2):**

| Что урезаем | Экономия |
|---|---:|
| L3 нагрузочные сценарии: 3 → 2 (убираем spike) | ~0.3 дня |
| L3 guardrails: `llm-guard` без дополнительной обёртки-метрик | ~0.3 дня |
| L1 Grafana: 2 дашборда → 1 комбинированный | ~0.2 дня |
| L2 MLflow: только текстовые трассы, без файловых artifacts | ~0.2 дня |
